## Repository shape
- `src/` — bot (`tg_bot.py`, `entrypoint.py`; UI/routing only — series math lives in `finance/portfolio_series.py`), quant engine (`finance/`), report layer (`pdf_payload.py`, `premium_payload.py`, `premium_renderer.py`, `html_renderer.py`, `templates/`), LLM narrative (`ai_narrative.py`), advisory audit (`agent/gatekeeper.py`).
- `tests/` — pytest/unittest suites (`test_phase*.py`); validation surfaces reviewed alongside code changes.
- `cloud_function/` — GCS-triggered RAG ingest (ChromaDB build from bank PDFs). Two-bucket flow: PDFs land in **INBOX** `ramp-bot-chroma-db-inbox-investadv` (the Cloud Function trigger, **path A**); the built ChromaDB lives in **STORE** `ramp-bot-chroma-db-investadv` and the bot mirrors it on boot (`entrypoint._download_chroma_db`). **Reliable fallback (path B, 2026-07-04; incremental since 2026-07-09):** `entrypoint._boot_ingest_from_inbox` — on boot the bot ingests any INBOX PDFs **not yet in the store** in-container on a daemon thread (deps + pre-baked ONNX model + pinned chromadb present) and republishes to STORE, bypassing the fragile Cloud-Function/Eventarc/region chain. It self-heals BOTH an empty store AND incremental uploads (compares INBOX basenames vs. ingested `source`s), so the CoVe «база: N отчётов» count recovers on the next restart even when the Cloud Function silently failed — the root cause of «счётчик не обновился после 8 новых отчётов». It ALSO **purges tmp-named artifact chunks** on boot (`rag_engine.purge_temp_sources`): pre-2026-07-05 ingests stored the `source` as a `tmpXXXX.pdf` basename, and re-ingests under real names left those behind → the «отчётов» count over-counted (48 shown vs. 29 real; `_is_temp_source` filters them from `list_documents` too). (`RAG_BOOT_INGEST=0` to disable; `cloud_function/rag_engine.py` is kept identical to `src/agent/rag_engine.py`; the function's `chromadb` is pinned `==0.6.3`). `scripts/ingest_bank_report.py` — admin CLI for fast/manual PDF ingest (`--upload` publishes ChromaDB to STORE).
