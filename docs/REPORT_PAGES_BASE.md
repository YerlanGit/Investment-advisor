> числа, как считается математика и как читать результат.** Парный файл по DEEP-тиру —

> посекционный аудит живых отчётов — `REPORT_SECTIONS_AUDIT.md`.

## Поток данных

Один и тот же конвейер для обоих тиров:

```
analyze_all()                      движок — src/finance/investment_logic.py
  (UniversalPortfolioManager)      считает risk_matrix, portfolio_metrics, asset_scores,
                                   regime, stress_scenarios, black_litterman, expected_effect …
        │
        ▼

        │   risk_waterfall, integrity_checks, ai_ideas …)
        ▼
Jinja-шаблон  src/templates/report_basic_v3.html   (+ партиал _mandate_compliance.html)
        │   рисует HTML/SVG, читает payload как data.<ключ>
        ▼
HTML по подписанной ссылке (Cloud Run отдаёт по signed URL)
```

ИИ-поля (`ai_*`) приходят **отдельной веткой**: `generate_narrative()` (`src/ai_narrative.py`,
модель `claude-sonnet-4-6` для BASE) → `ai_summary` → те же payload-ключи. Любой текст в
блоках «ИИ-комментарий» — **advisory (пояснение), а не индивидуальная инвестиционная
рекомендация**; цифры берутся из движка, ИИ их только комментирует. Когда API недоступен,
ИИ-блоки просто скрываются (`{% if data.ai_* %}`), а отчёт остаётся информативным.

> **Важно про шаблон.** В шапке файла стоит пометка «ПРОТОТИП (design-only)». Числовые
> поля давно подключены к боту через `build_payload`; «прототипными» остаются только
> декоративные SVG (equity-curve линия на стр. 3 и спарклайны KPI — последние спрятаны за

# Лист 1 — Обложка · KPI · концентрация риска

## 1.1 Тех-хедер

**Назначение.** Машинная «шапка» каждого листа: идентификатор запуска.

**Рендерит:** `data.tier` (→ `BASE`), `user_id`, `generated_at`. Строка вида
`PORTFOLIO RISK REPORT // TIER: BASE // ID: U<user_id> // RISK_ENGINE: MAC3 … GENERATED <время>`.
Литерал `MAC3` — имя квант-движка (MAC3RiskEngine).

**Откуда.** `tier` и `user_id`/`generated_at` прокидываются в шаблон при рендере; `tier`
в payload кладёт `build_payload` (последний ключ словаря).

**Как читать.** Технический штамп — на содержание не влияет.

## 1.2 Вердикт по портфелю (H1 + резюме + буллеты ИИ)

**Назначение.** Главный заголовок отчёта — одно предложение «что в целом с портфелем»,
плюс короткое пояснение простым языком и список инсайтов.

**Payload-ключи:**
- `data.ai_verdict` — H1 (≤1 предложения). Если пусто — статический фолбэк: «Портфель в зоне
  `{{ risk_label }}` риска — ключевые метрики и концентрация риска ниже».
- `data.ai_plain_summary` — текст в голубом ai-box (≤2 предложений, простой язык). Фолбэк
  объясняет, что такое индекс риска, и подставляет `data.risk_pct` / `data.risk_label`.
- `data.ai_bullets[]` — компактный список инсайтов; BASE рендерит **первые 4** (`ai_bullets[:4]`).
- `data.used_rag` — если True, в фолбэк-резюме добавляется отметка `[RAG: Bank reports]`.

**Builder / источник.** Чистая прокидка в `build_payload`: `ai_verdict ← ai_summary["verdict"]`,
`ai_plain_summary ← ai_summary["plain_summary"]`, `ai_bullets ← ai_summary["bullets"]`. Сам текст
рождает `generate_narrative()` (`ai_narrative.py`) — структурированным tool-вызовом `emit_report`.

**Как читать.** H1 — короткий человеческий вывод. Зелёный/янтарный тон ниже зависит от
числовых метрик, а не от формулировки. Текст ИИ — пояснение, **не рекомендация к сделке**.

> **Исторический контекст (аудит):** до Sprint 5.2 H1 был **захардкоженным** прототипным
> предложением, одинаковым для всех; `ai_bullets` оплачивались, но не рендерились нигде.
> Сейчас H1 привязан к `ai_verdict`, статика — только no-AI фолбэк (без выдуманных фактов).

## 1.3 Панель «Соответствие мандату»

**Назначение.** Показывает целевой профиль инвестора (мандат) и насколько текущий портфель
ему соответствует: по каждому классу активов — допустимый диапазон и фактический вес.

**Включается** партиалом `{% include "_mandate_compliance.html" %}` внутри `cover-main`
(после комментариев ИИ — завершает блок «Вердикт по портфелю»). Скрыта, если профиля нет.

**Payload-ключ:** `data.mandate_compliance` со структурой
`{profile_name, target_vol_pct, target_te_pct, rows[], breaches, compliant, leveraged, margin_debt_pct}`.
Каждая строка `rows[]` = `{key, label, actual, lo, hi, status}`.

**Builder:** `_build_mandate_compliance(perf_df, total_val, user_profile, leverage)` (`pdf_payload.py`).

**Как работает логика:**
1. Фактическое распределение по классам: для каждой позиции `cv/total_val*100`, агрегируется по
   ключу класса через `agent.gatekeeper._classify_to_asset_key` (кэш → `Cash`, не `Bonds`).
2. По каждому лимиту `[lo, hi]` из `user_profile["limits_dict"]`: с допуском ±2 п.п. —
   `actual > hi+2` → `over` (это нарушение, `breaches += 1`); `actual < lo-2` → `under` (недовес,
   нарушением **не** считается); иначе `ok`.
3. **Плечо.** Если книга на марже (`leverage.is_leveraged`), считается `margin_debt_pct =
   |min(0, cash_weight)|·100`. Леверидж-книга **по построению вне лимитов классов** ⇒
   `compliant = (breaches == 0 and not leveraged)`.
4. Рендер партиала: для каждой строки рисуется мини-бар — затенённая полоса `[lo, hi]` и маркер
   на `actual` (цвет по `status`: ok/under/over). Отдельной пунктирной строкой — «Маржинальный
   долг −X% · вне лимитов классов».

**Как читать.** Чип `✓ в рамках` (зелёный) или `⚠ нарушений: N` (красный). Маркер выше
правой границы полосы = перевес класса; ниже левой = недовес. Маржинальный долг всегда вне
лимитов классов, поэтому даже один такой пункт делает портфель не-compliant.

## 1.4 Риск-гейдж 0–100 + мандат-бейдж

**Назначение.** Полукруглый «спидометр» сводного индекса риска и подпись, для какого профиля
он откалиброван.

**Payload-ключи:** `data.risk_pct` (0–100, целое), `data.risk_label`
(`Консервативный`/`Умеренный`/`Агрессивный`), `data.risk_mandate_label` (имя профиля под гейджем).

**Builder.** Верх `build_payload`: `risk_pct = clamp(Composite_Risk_Score, 0, 100)`,
`risk_label = _risk_score_label(risk_pct)` (<33 / <66 / иначе). `risk_mandate_label` = реальное
имя профиля (`user_profile["profile_name"]`), иначе русское имя мандата `_risk_mandate_label`.

**Движок — как считается composite_risk_score** (`finance/scoring.py::composite_risk_score`):
сводный 0–100 индекс — это **взвешенный по мандату бленд трёх нормированных сигналов**:

```
s_vol  = clamp(vol / 0.40, 0..1) · 100          # 40% годовой волатильности = 100
s_cvar = clamp(|CVaR| / cvar_base, 0..1) · 100  # cvar_base зависит от мандата
s_conc = clamp(maxERC% / 50, 0..1) · 100        # 50% концентрации одной позиции = 100
score  = w_vol·s_vol + w_cvar·s_cvar + w_erc·s_conc   (round → int)
```

Веса и база CVaR — из матрицы мандатов `_RISK_MANDATE_MATRIX` (каждая строка весов = 1.0):

| Мандат | w_cvar | w_vol | w_erc | cvar_base |
|---|---|---|---|---|
| CONSERVATIVE | 0.60 | 0.30 | 0.10 | 0.03 |
| MODERATE | 0.40 | 0.40 | 0.20 | 0.065 |
| AGGRESSIVE | 0.20 | 0.50 | 0.30 | 0.08 |

Консерватор более чувствителен к хвосту (выше w_cvar и **уже** база CVaR ⇒ тот же хвост даёт
бо́льший индекс); агрессор терпит хвост (ниже w_cvar, шире база) и сильнее реагирует на голую
волатильность. Все входы прогоняются через `_finite()` (NaN/Inf → 0), чтобы один битый апстрим
не отравил индекс. Результат `clamp(0,100)` (аудит 06-14: иначе фолбэк `vol/0.40·100` на
вырожденной книге мог перекрутить стрелку за шкалу).

**SVG-геометрия гейджа.** Стрелка нормализована (указывает вверх) и поворачивается:
`rotate = (risk_pct − 50) · 1.8°` вокруг пивота (140,138). То есть 0 → −90°, 50 → вертикаль,
100 → +90°. Зоны на дуге: НИЗКИЙ / УМЕРЕННЫЙ / ВЫСОКИЙ.

**Как читать.** Стрелка влево/зелёное = низкий риск, вправо/красное = высокий. Подпись «Индекс
рассчитан для вашего профиля: <имя>» важна: один и тот же портфель даст разный индекс под разным
мандатом (другая база CVaR и веса), поэтому число читается в контексте профиля.

> **Аудит:** бейдж раньше брал 3-бакетную калибровку («Умеренный»), что противоречило панели
> мандата («Умеренно-агрессивный»). Фикс 5.2: при наличии профиля бейдж показывает реальное имя
> профиля; внутренняя шкала гейджа (бакеты) не тронута.

## 1.5 KPI-полоса: CVaR / Sharpe / MaxDD

**Назначение.** Три ключевые метрики риска крупным шрифтом, с долларовыми оценками и
ИИ-комментариями.

**Payload-ключи:**
- KPI-1 CVaR 95%: `data.cvar`, `data.cvar_dollar`, `data.cvar_ci`, `data.ai_cvar_note`.
- KPI-2 Sharpe: `data.sharpe`, `data.sortino`, `data.ai_sharpe_note`.
- KPI-3 Max Drawdown: `data.max_drawdown`, `data.mdd_dollar`, `data.ai_mdd_note`.
- (общий) `data.kpi_sparklines` — **[статический SVG, спрятан]**; движок пока не отдаёт
  по-KPI историю, блок скрыт за `{% if data.kpi_sparklines %}`.

**Builder.** KPI-блок `build_payload` форматирует строки из `portfolio_metrics`:
`cvar_str = f"{cvar_raw*100:.1f}%"`; `cvar_ci_str = f"{lo*100:.1f}% … {hi*100:.1f}%"` (голый
диапазон — слово «разброс» добавляет шаблон); `cvar_dollar = "${|cvar·total_val|:,.0f}"`. ИИ-ноты —
прокидка из `ai_summary` (`ai_cvar_note`/`ai_sharpe_note`/`ai_mdd_note`).

**Движок — математика метрик** (`investment_logic.py`):

- **CVaR 95% (1 день).** Точечная оценка + 95% CI считаются bootstrap-ом `_bootstrap_cvar`:
  *стационарный блок-bootstrap* с геометрической длиной блока (среднее ≈ √N), сохраняющий
  автокорреляцию дневных доходностей (наивный iid недооценивает хвост). На каждой из 2000
  ресемплированных серий: `VaR_b = перцентиль(sample, 5%)`, `CVaR_b = среднее хвоста
  {r ≤ VaR_b}`. Итог: `point = mean(CVaR_b)`, `lo95/hi95 = перцентили 2.5%/97.5%`. CI **асимметричен**
  — признак настоящего bootstrap, а не `±k·σ`. Сид детерминирован от самих данных (round-6 хэш):
  одинаковый вход → один и тот же CI, но при сдвиге окна CI «дышит».
  *Читать:* «худшие 5% дней теряют в среднем X% за день»; модуль больше = тяжелее хвост.

- **Sharpe / Sortino.** Годовые, на trailing дневных лог-доходностях в валюте отчёта.
  `Sharpe = (ann_return − RFR) / σ_ann`. `Sortino = (ann_return − RFR) / downside_vol`, где
  `downside_vol = √(mean(min(excess,0)²)) · √252` — индустриальная *downside deviation* (шортфоллы
  против MAR=0, усреднённые по **всем** N наблюдениям, не только по убыточным дням). RFR годовой
  echo-ится движком (env `KZ_RFR_ANNUAL`, дефолт 0.14) с **геометрическим** дневным переводом
  `(1+r)^(1/252)−1`. *Читать:* Sharpe >1 хорошо, ~0.5 средне, <0 — доходность не покрывает
  безрисковую ставку. Sortino ≥ Sharpe (наказывает только падения).

- **Max Drawdown.** Максимальная просадка пик→дно на equity-кривой; модуль и `$`-оценка.
  *Читать:* «худший зафиксированный спад от вершины»; глубже = больнее переживался портфель.

**Цвет рамки KPI-карт** статический CSS-класс: CVaR — `st-normal` (зелёная кромка), Sharpe —
`st-good` (синяя), MaxDD — `st-watch` (янтарная). (Эти кромки декоративны и не зависят от значения.)

**ИИ-ноты.** Поясняют каждую метрику простым языком — advisory.

> **Аудит (06-14, ✅):** $-оценки KPI сходятся с NAV; CI асимметричен (реальный bootstrap);
> числа идентичны между BASE и DEEP — один движок, паритет.

## 1.6 «Где сосредоточен ваш риск» — таблица TRC

**Назначение.** Топ-5 позиций по вкладу в риск портфеля: видно, где «спрятан» риск, даже если
вес позиции невелик.

**Payload-ключ:** `data.assets[]` — поля `ticker`, `weight`, `beta`, `euler_risk`, `hotspot`.
Шаблон сортирует `assets | sort(attribute='euler_risk', reverse=true)` и берёт `[:5]`.

**Builder.** Per-asset цикл в `build_payload`: `euler_risk = f"{euler:.1f}%"` из колонки
`Euler_Risk_Contribution_Pct`, `beta` из `Beta_Market` (NaN-safe через `np.isfinite`), флаг
`hotspot` приходит из движка (НЕ переоценивается в шаблоне — комментарий H-2).

**Движок — Эйлер-TRC** (`investment_logic.py::calculate_structural_risk`). Сначала строится
**структурная факторная ковариация**: для каждого актива Ridge-регрессия (α=0.001) дневных
лог-доходностей на фактор-ETF даёт беты B и специфическую дисперсию D; факторная ковариация —
бленд EWMA (halflife=63 дн, λ≈0.99) 70% + Ledoit-Wolf 30%; затем
`Σ = B·F·Bᵀ + D`, годовая `Σ_ann = Σ·252`. Декомпозиция Эйлера:

```
σ_p   = √(w' Σ_ann w)                 # волатильность портфеля
MCTR  = (Σ_ann · w) / σ_p             # маржинальный вклад актива в риск
ERC%ᵢ = wᵢ · MCTRᵢ / σ_p · 100        # доля актива в полном риске (Σ ERC ≈ 100%)
```

**HOTSPOT-порог.** `🔥`/`HOTSPOT` ставится, когда `ERC% > scoring.HOTSPOT_TRC_PCT` (= **20.0**) —
**единственное место тюнинга** (используют и отчёт, и Gatekeeper GK-1). Шаблон рисует пилюлю
`HOTSPOT` (янтарь) при `a.hotspot`, иначе `NORMAL` (зелёный).

**Как читать.** Высокий «ВКЛАД В РИСК %» при умеренном «ВЕС %» = позиция тянет риск
непропорционально (высокая бета или корреляция). HOTSPOT = одна бумага собрала >20% всего риска
портфеля — кандидат на частичную фиксацию. BETA — чувствительность к рынку (>1.5 — высокая).

## 1.7 Риск-водопад (разложение волатильности)

**Назначение.** Наглядно: насколько диверсификация «съедает» суммарный риск отдельных позиций.

**Payload-ключ:** `data.risk_waterfall` — `{bars[], sum_standalone_pp, diversification_pp,
total_vol_pp, y_max_pp, …}`. Каждый бар: `ticker`, `standalone_pp`, `cum_end_pp`.

**Builder:** `_build_risk_waterfall(risk_matrix, perf_df, total_val, total_vol_ann)` (`pdf_payload.py`).

**Как работает математика** (всё в годовых десятичных, 0.142 = 14.2%):

```
σᵢ            = √Σᵢᵢ                  # годовая волатильность актива (диагональ ковариации)
standaloneᵢ   = wᵢ · σᵢ              # вклад «как если бы актив был один»
Σ standalone  = Σᵢ (wᵢ · σᵢ)        # недиверсифицированный риск (сумма «в столбик»)
diversified   = √(w' Σ w)           # реальный риск портфеля (= total_vol_ann)
benefit       = Σ standalone − diversified   ≥ 0   (эффект диверсификации)
```

Неотрицательность benefit гарантирована неравенством Минковского `√(w'Σw) ≤ Σ|wᵢ|·√Σᵢᵢ` для
неотрицательных весов. Builder также готовит **геометрию** (это была математика-в-Jinja, теперь
вынесена в backend): топ-4 бара + агрегат «+N др.», кумулятивные офсеты `cum_end_pp`, динамический
домен оси Y `y_max_pp = ceil(max(Σstandalone, total, 20)/5)·5`. Шаблон только мапит п.п. → пиксели
(`y_scale = 147 / y_max_pp`) — чистое view-масштабирование.

**Что на графике.** Янтарные бары (риск каждой позиции отдельно) набираются стопкой вверх →
зелёный шаг вниз «диверсификация» (−benefit) → синий бар «итоговый риск» (полная
диверсифицированная волатильность от нуля). Пустое состояние — «нужна ковариационная матрица
(≥30 дней истории)».

**Как читать.** Чем больше зелёный шаг относительно янтарной стопки, тем лучше диверсификация
гасит риск. Маленький зелёный шаг = позиции сильно скоррелированы (скрытая общая ставка).

## 1.8 Контроль качества данных (integrity)

**Назначение.** Прозрачность: какие источники реально использованы на этом запуске и в каком
состоянии (свежесть/покрытие).

**Payload-ключ:** `data.integrity_checks[]` — список `{status, label, detail}`. Один и тот же
блок повторяется **внизу каждого листа** BASE-отчёта (стр. 1–4).

**Builder:** `_build_integrity_checks(results, ai_summary, data_quality, return_series_coverage)`
(`pdf_payload.py`). Состояния: `✓` (данные есть, в окне свежести), `⚠` (частично/деградация),
`—`/`✗` (источник не использован/недоступен).

**Что проверяется (последовательно):** рыночные данные (Tradernet, N дней); серия доходностей
(покрытие % / исключённые тикеры из sparse-history); факторная модель (Ridge β, M/10 факторов,
% покрытия — ✓ при ≥80%); Euler-декомпозиция (`TRC = w·MCTR/σ_p`, Ledoit-Wolf 70/30); CVaR
Bootstrap (Politis-Romano, 2000 блоков, 95% CI — ✓ при наличии CI); RAG банк-отчётов (3-state:
used `✓` / no_match `—` / unavailable `✗`; **+ инвентарь базы** «M отчётов · K чанков», 2026-07-04);
**ИИ↔банк-аналитика** (2026-07-04: `⚠` когда нарратив ссылается на банки при пустой базе — цитаты
из знаний модели, не подтверждены отчётами; `✓` при проверенных `[RAG]`-цитатах); AI-модель
(имя или «не задействована»); валюта отчёта + RFR (провенанс); FX-конверсия (только если что-то
конвертировалось, ⚠ при T-1 фолбэке).

**Как читать.** Зелёные `✓` — всё чисто. `⚠` — стоит знать (частичное покрытие/устаревший
кэш). Это не оценка портфеля, а самоаудит данных, на которых построен отчёт.

# Лист 2 — Что вы держите (holdings)

## 2.1 Таблица позиций + раскрытие SEC-фундаментала

**Назначение.** Полный список позиций с весом, риском, P/L и действием; по клику строка
раскрывает фундаментал бумаги (SEC EDGAR) и ИИ-комментарий.

**Payload-ключи.** Заголовок: `data.holdings_count`. Цикл `data.assets[]` — поля `ticker`,
`asset_class`, `weight`, `euler_risk`, `pnl_pct`, `pnl_abs`, `pnl_color`, `action`, `is_cash`,
`weight_pct_num`, `hotspot`, `atr_pct`, `atr_extreme`, `extremes[]`, `ai_comment`. Фундаментал
джойнится по тикеру из `data.fundamental_layer[]` (`roe`, `op_m`, `dta`, `rev_g`, `altman_z`).

**Builder.** Per-asset цикл `build_payload`:
- `asset_class` — из брокерской метки `Asset_Class_Label`, фолбэк — `classify_asset_class(ticker)`
  (SSOT в `finance/scoring.py`, suffix-aware). `is_cash = (asset_class == "Ден. средства")`.
- `weight = f"{weight_pct:.1f}%"`, `weight_pct = cur_val/total_val·100`.
- `pnl_pct = f"{ret_pct*100:+.1f}%"`, `pnl_abs = f"{pnl:+,.0f}"`; `pnl_color` = `neut` для кэша,
  иначе `pos`/`neg` по знаку `ret_pct`.
- Кэш **никогда** не несёт скор, action или hotspot. Отрицательный вес кэша = маржинальный долг
  → бейдж «Долг» (пунктир, нейтраль), а не «прибыль».
- `extremes[]` — словесные флаги: «вес > 15%», «TRC > 20%», «ATR > 3%» (пороги `EXTREME`, функция
  `_flag`).
- `fundamental_layer` — SEC-колонки (`SEC_ROE`, `SEC_Op_Margin`, `SEC_Debt_to_Assets`,
  `SEC_Revenue_Growth_YoY`, `SEC_Altman_Z`…), форматируются `_fmt_pct`/`_fmt_num`. Тикеры без SEC
  (сектор `default`/`EM_Proxy`) пропускаются → в раскрытии «н/д».

**ИИ-комментарий в раскрытии.** Если `asset.ai_comment` пуст, шаблон даёт **rule-based фолбэк**
по приоритету сигнала: (1) HOTSPOT → контекст концентрации + TRC + P/L; (2) `atr_extreme` →
повышенная внутридневная волатильность; (3) `pnl_color=neg` → контекст просадки; (4) прочие
`extremes` → список триггеров; (5) дефолт — «вклад умеренный, удерживаем». Цитата-источник
`[SEC EDGAR] [Quant Engine]` (+ имя модели, если коммент от ИИ).

**Как читать.** «Риск» в строке = тот же Euler-TRC (ERC%). HOTSPOT-флаг рядом с тикером =
концентрация >20%. ATR красным = бумага «дёргается» сильнее среднего. Бейдж действия: BUY
(зелёный) / HOLD (серый) / TRIM (янтарь) / SELL (красный). В раскрытии — фундаментал по
последней годовой отчётности; «н/д» = SEC-данных нет (ETF, сырьё, EM-прокси).

## 2.2 Панель секторов + пай + плечо

**Назначение.** Структура портфеля по секторам (пай), бейдж плеча/gross-exposure и
предупреждения о перевесе/мандате.

**Payload-ключи:** `data.pie_chart_data` (приоритет) или `data.sectors` — `{name, weight_pct,
weight_str, nav_pct}`; `data.leverage_metrics` (`is_leveraged`, `gross_exposure`,
`leverage_ratio`, `cash_weight`); `data.sector_warnings[]` (`text`); `data.ai_leverage_warning`.

**Builder.** Sector-блок `build_payload`:
- Под плечом длинные сектора суммируются к **>100% NAV** (когда отрицательный кэш-маржа фондирует
  лонги). Чтобы легенда не показывала 118%, пай нормализуется к **доле длинной книги** (`w/_long_sum`),
  сумма ровно 100%; `nav_pct` хранит сырую NAV-долю. Плечо при этом показано отдельным бейджем.
- `leverage_metrics` — из движка: `gross_exposure = (лонг+|маржа|)/капитал`,
  `leverage_ratio = лонг/капитал` (подписи баз L1).
- `sector_warnings` — `_build_sector_warnings` (порог `SECTOR_CAP_PCT = 40%`) + супергруппы
  `_sector_group_warnings`. **SSOT супергрупп** `SECTOR_SUPERGROUPS` («Tech-комплекс =
  Technology+Semiconductors») гарантирует, что заголовочная «tech-концентрация» — ОДНО
  авторитетное число (а не ad-hoc пересчёт ИИ).

**Геометрия пая.** Шаблон строит `conic-gradient` инлайн через namespace: накапливает проценты,
вставляет 1% белый разделитель между секторами; палитра `[amber, tech-blue, green, ink-60, ink-40]`
— крупнейший сектор красится янтарём (alert). Сектор >40% подсвечивается в легенде (`over`).
Бейдж плеча показывается **только** при `is_leveraged` (cash_weight < −0.001), формула с подписями.
`ai_leverage_warning` (Margin-Call) — красная карточка, заполняется ИИ только на маржинальной книге.

**Как читать.** Один сектор >40% (янтарь) = концентрация по сектору. «Gross Exposure 131% ·
Плечо 1.16x» = на каждый $1 капитала работает $1.31 экспозиции, маржа усиливает и прибыль, и
убыток. Warnings перечисляют конкретные превышения (включая Tech-комплекс).

> **Аудит (✅):** «Gross 131% / 1.16x» с подписями баз работает вживую; пай нормализован к лонгу,
> числа взаимно согласованы (`1.16 ≈ 1.31 − 0.16 + 0.01`). До фикса ИИ на BASE писал «прибыль
> удваивается» при 1.16x — `leverage_rule` теперь запрещает «удваивается», если плечо не ≈2x.

## 4.2 Сетка идей (4 категории, конвейер Factor→Regime→RAG)

**Назначение.** До 4 стратегических идей-карточек (рост · ребалансировка · защита · Smart Money), с
тикерами-кандидатами **вне портфеля** и раскрываемым обоснованием (конвейер отбора).

**Payload-ключи:** `data.ai_ideas` (dict бакетов) + `data.ideas_count`. Шаблон уплощает бакеты
в порядке `['risk_reduction', 'diversification', 'growth', 'rotation', 'quality', 'hedge']`.
Каждая идея: `category`, `priority` (high/medium/low), `idea_num`, `title`, `rationale`,
`candidates[]` (`{ticker, name, scenario}`), `pipeline[]` (стадии), `sources[]`.

**Builder:** `_build_ai_ideas(stock_picks, tier="base")` (`pdf_payload.py`). Маппит
scenario-ключи ИИ `boost_alpha/rebalance/protect_capital/smart_money` → бакеты
`growth/diversification/hedge/rotation` с русскими категориями и приоритетом (4-я карточка —
**Smart Money**, институционалы+инсайдеры — рендерится в bucket `rotation`). **Конвейер**
синтезируется из типа пика (`_PIPELINE_BASE`) — 3 стадии:
- `boost_alpha`: FACTOR (Momentum+Quality) → REGIME (соответствие режиму роста) → RAG (банк-отчёты);
- `rebalance`: FACTOR (4-Pillar F+V+T+C) → REGIME (секторная ротация) → RAG (фундаментал);
- `protect_capital`: FACTOR (низкая Beta+дивиденды) → REGIME (защитные сектора) → RAG (хвостовой риск);
- `smart_money`: FACTOR (сигнал умных денег: институц. накопление) → REGIME (позиционирование фондов/инсайдеров под режим) → RAG (13F-фонды + инсайдеры SEC Form 4).

Если ИИ не вернул `label` (или label == raw-ключ), подставляется каноническое русское имя
категории (фикс 5.2 — иначе в заголовках текли сырые ключи `boost_alpha`).

**Источник пиков.** `ai_summary.stock_picks` от `ai_narrative._user_prompt`: правило
**DATA-DRIVEN** (привязка к портфелю) + **СВЕЖЕСТЬ ИДЕЙ** (дневной срез `YYYY-MM-DD` +
дневной «угол ротации» из `_IDEA_ROTATION_ANGLES`, запрет повтора «по привычке»). Фильтры: `_remove_held_picks` (запрет уже имеющихся тикеров) →
`_check_pick_contradictions` → `_backfill_empty_scenarios` (фолбэк-каталог помечается
«· резервный каталог»). Без API — `_fallback_stock_picks` (месячная ротация). BASE: модель
`claude-sonnet-4-6`, `temperature≈0.7` (env `ANTHROPIC_TEMPERATURE`, band 0.5–0.85).

**Рендер карточки.** Цветовая полоса по категории; чип приоритета; свёрнутая карточка показывает
title + rationale + чипы тикеров-кандидатов; по клику раскрываются конвейер (Factor→Regime→RAG),
разбор «почему именно эти бумаги» (по кандидатам), ожидаемый эффект (если есть), источники.
Пустое состояние — «Идеи появятся после генерации AI-рекомендаций».

**Как читать.** Тикеры в карточке — **не из вашего портфеля**: ИИ подбирает их под смысл идеи
(замена/дополнение). Приоритет high/medium/low — насколько идея «горящая». Конвейер показывает,
через какие фильтры кандидат прошёл. Всё это — **аналитические идеи, не рекомендация**.

> **Аудит:** на BASE Haiku тратил пики на уже-имеющиеся тикеры, held-фильтр их вычищал, бэкфилл
> подставлял каталог без пометки. Фиксы 5.2: held-rule в промпте + пометка «резервный каталог» +
> каноническое имя категории вместо сырого ключа.

## 4.3 Дисклеймер и полоса источников

**Назначение.** Юридическая дисклеймер-плашка и перечень источников данных.

**Как читать.** Главное: отчёт — **аналитический материал, не ИИР** (индивидуальная
инвестиционная рекомендация). Источники перечислены для аудита.
