## Ключевое архитектурное решение: ZERO LLM-API бэктест

Бэктест — **чистая детерминированная математика на уже загруженных данных**. LLM в нём не
нужен вовсе (опционально — один дешёвый вызов для резюме). Стоимость на отчёт ≈ $0.00–0.01.

```
Кнопка «Проверить идею историей» (Action Plan / карточка идеи)
   │  payload: {ticker|basket, action, даты, параметры правила}
   ▼
scenario_engine.replay()  ← НОВЫЙ модуль src/finance/scenario_engine.py (sklearn-free)
   │  вход: те же дневные CLOSE, что уже в кэше истории (+§lookback-план до 1260д)
   │  логика: walk-forward по историческим датам:
   │    1. на каждую дату t из сетки (напр. раз в месяц) пересчитать 4-Pillar
   │       сигнал/правило идеи НА ДАННЫХ ДО t (никакого look-ahead);
   │    2. если правило срабатывало — виртуальная сделка по close(t+1) (без
   │       внутридневных допущений), горизонты 1M/3M/6M;
   │    3. собрать распределение исходов: hit-rate, средний/медианный P&L,
   │       worst case, vs держать SPY.
   ▼
scenario_payload → отдельный HTML-тир (шаблон = облегчённый premium, 1 стр.)
```

## Почему это дёшево и не ломает текущую логику

- **Данные:** ни одного нового сетевого вызова — история уже тянется для основного отчёта
  (кэш `/tmp/freedom_history_cache`); при расширении lookback (см. ROADMAP_DATA_RESILIENCE §4)
  качество бэктеста растёт автоматически.
- **Код:** новый модуль — аддитивный; `scoring.py` уже чистые функции от DataFrame →
  переиспользуются as-is с усечённым по дате фреймом (главное правило: **передавать
  `df[df.index <= t]`**, ничего внутри scoring не менять).
- **LLM:** ноль обязательных вызовов. Опция: 1 вызов Haiku (≈$0.003) для абзаца-резюме.
- **Токеномика:** тир тарифицируется дешевле DEEP (пример: 1 токен), т.к. нет LLM-затрат.

## Жёсткие ограничения (сказать пользователю в отчёте)

1. **Survivorship bias** — идеи тестируются на живых сегодня тикерах; результат завышен.
2. **730д (сейчас) — слишком коротко** для statistically significant hit-rate: <24 месячных
   точек. Полноценный тир требует lookback ≥1260д (см. Data Resilience §4) — внедрять ПОСЛЕ.
3. Без комиссий/проскальзывания в v1 (добавить константой 10 б.п. на сделку в v2).
4. Это **иллюстрация правила, а не гарантия** — тир не превращает советы в сигналы.
