## 1. Экономика DEEP-отчёта: где на самом деле деньги

Профиль DEEP-вызова: ~25–30K токенов вход (25KB system — **уже под prompt-cache 0.1×**,
данные+RAG ~10–15K) + 5–6K выход, 1 вызов (Structured Outputs). Фактические $0.30–0.70 —
**доминирует ВЫХОД** (Opus $25/M out) и некэшируемая часть входа.

| Вариант (модель DEEP) | Вход/Выход $/M | DEEP-отчёт ≈ | Латентность ≈ | Качество RU-финпрозы | Интеграционная цена |
|---|---|---|---|---|---|
| **Opus 4.8 (текущий)** | 5 / 25 | $0.30–0.45 (с кэшем) | 40–60 c | эталон | 0 (есть) |
| **Sonnet 4.6** | 3 / 15 | $0.15–0.25 | 20–35 c | −5–10% (проверить A/B) | 0 (один SDK, флаг `ANTHROPIC_MODEL_DEEP`) |
| GPT-4o / Gemini 1.5 Pro (запрошены) | — | — | — | — | **устаревшие поколения (2024); сравнивать с ними в 2026 некорректно** |
| OpenAI флагман (актуальный, GPT-5-класс) | ~2.5–10 / 10–30 | сопоставимо ±30% | 25–50 c | сильное EN, RU — проверять | **ВЫСОКАЯ**: переписать Structured Outputs (наш forced `tool_choice` — Anthropic-специфичен), prompt-cache API другой, guard-фильтры перекалибровать |
| Gemini (актуальный Pro-класс) | ~1.25–5 / 5–15 | ниже ±40% | 20–40 c | RU ок, KZ лучший из трёх | та же ВЫСОКАЯ + другой JSON-режим |

### Жёсткие выводы CTO
1. **Главная экономия НЕ в смене вендора.** Переход DEEP Opus→**Sonnet 4.6** даёт −40–50%
   стоимости и −40% латентности **одной строкой env** (`ANTHROPIC_MODEL_DEEP=claude-sonnet-4-6`),
   без единого риска интеграции. Сначала A/B 20 отчётов (чек-лист: полнота 15 полей,
   выполнение лимитов, [RAG]-цитаты, стресс-кап валидатор) — потом решение.
2. **Мульти-вендорный выбор в боте (Claude/GPT/Gemini) — отклонить в текущем виде.**
   Скрытая цена: 3 SDK × (structured-output диалекты, кэш-API, retry-семантика),
   guard-пайплайн (held/противоречия/стресс-кап/CoVe-цитаты) калиброван под Claude —
   каждая модель = собственная матрица регрессий; тестовая поверхность ×3 при команде в
   1 человека. **Альтернатива с той же UX-ценностью:** переключатель «⚡ Быстрый (Sonnet) /
   🔬 Глубокий (Opus)» — один SDK, честная цена/скорость, реализация ~1 день
   (меню → env-override per-request).
3. Дальнейшая оптимизация по убыванию ROI: (а) сокращение НЕкэшируемого входа (данные-скелет
   уже ужат `_summarise_for_prompt`); (б) Batch API для отложенных отчётов (−50%), если
   появится «отчёт к утру»; (в) стриминг для UX (не денег).

## 2. Кросс-язычный RAG (EN-PDF ↔ RU/KZ) — строгий разбор

### Текущее состояние (факты из кода)
- Эмбеддинги: ChromaDB `DefaultEmbeddingFunction` = **all-MiniLM-L6-v2** — англоцентричная,
  384-dim, RU понимает слабо, **казахский — практически нет** (домен отсутствует в трейне).
- **Спасительный дизайн-факт:** все три запроса `_fetch_rag_context` (macro/micro/regime) —
  **захардкожены на английском** → retrieval сегодня работает EN→EN, деградации НЕТ.
  Это надо зафиксировать как инвариант (комментарий в коде + тест «запросы латиницей»).
- Перевод смысла: EN-чанки → RU-нарратив делает Opus в основном вызове; точность терминов
  (duration, spread, breakeven) высокая, но НЕ проверяется автоматически.

### Риски (по убыванию)
| Риск | Вероятность | Эффект |
|---|---|---|
| Кто-то «улучшит» запросы на русский → recall падает на 30–60% (MIRACL-класс бенчмарки для EN-моделей) | высокая (без теста-инварианта) | тихая деградация RAG |
| KZ-выдержки/KZ-ответы: MiniLM не решает, casual-перевод LLM терминов (қамтамасыз ету etc.) нестабилен | средняя | галлюцинации терминологии |
| Перевод чисел/знаков в нарративе (b.p. ↔ п.п.) | низкая (CoVe/валидаторы частично ловят) | искажение фактов |

### Решения (по этапам, без ломки текущего)
1. **Сейчас (0 затрат):** закрепить EN-запросы тестом-инвариантом; в чанк-метаданные писать
   `lang` (детект по алфавиту при ингесте) — основа для будущего фильтра.
2. **RU-запросы пользователей (когда появятся):** query-translation RU→EN одним вызовом
   Haiku 4.5 (≈$0.0002, +300–500 мс) ПЕРЕД векторным поиском — сохраняет EN→EN retrieval;
   fallback при таймауте — исходная строка.
3. **Смена эмбеддингов (при KB >100 документов или KZ-контенте):** локальная
   **bge-m3 / multilingual-e5-large** (0 API-затрат, RU/KZ приемлемо, нужен re-ingest всей
   базы + рост образа ~1–2 ГБ) ИЛИ Cohere embed-multilingual-v3 / OpenAI
   text-embedding-3-large (API-затраты ~$0.02–0.13/M ток., но новый вендор+ключ).
   Решение принимать ТОЛЬКО с бенчмарком: 30 эталонных вопросов RU/KZ ↔ признанные
   релевантными чанки, recall@5 до/после.
4. **KZ-выход:** честная позиция — пока НЕ обещать казахский в отчётах; при запросе —
   отдельный проход «перевести готовый RU-отчёт» (Sonnet, +$0.05) с глоссарием
   фин-терминов KZ в system-промпте; НЕ генерировать KZ напрямую из EN-чанков.

### Шаги внедрения (этап 1–2)
| # | Шаг | Оценка |
|---|---|---|
| 1 | Тест-инвариант «RAG-запросы — EN» + `lang` в метаданных ингеста | 0.5 дн |
