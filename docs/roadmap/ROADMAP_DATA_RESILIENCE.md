## 1. SPOF: fallback-провайдер цен (Tradernet ↓ → yfinance)

**Факт:** цены берутся ТОЛЬКО из Tradernet (`freedom_portfolio/history.py`); yfinance был
намеренно удалён 2026-04. При сбое брокера продукт слепнет целиком.

**Архитектура (аддитивная, за флагом):**
```
finance/price_providers.py (НОВЫЙ)
  PriceProvider (Protocol): fetch(tickers, days) -> HistoryResult
  TradernetProvider  — обёртка текущего history.py (поведение 1:1)
  YFinanceProvider   — lazy import yfinance; маппинг тикеров AAPL.US→AAPL,
                       FFSPC….AIX → НЕТ на Yahoo (честный partial-fallback!)
  ChainProvider      — primary→fallback per-ticker: тикеры, которые Tradernet
                       не отдал за N ретраев, добираются из fallback; source
                       фиксируется В КАЖДОЙ колонке (для CoVe)
```
**Критично (Do-No-Harm):** склейка двух источников в одну ковариацию опасна
(разные Close-конвенции: сплиты/дивиденды, часовые пояса, валюты листинга).
Правила: (а) fallback ТОЛЬКО целиком-по-тикеру, не по-дневно; (б) adjusted close
yfinance → привести к «total-return близкой» конвенции или ЯВНО пометить;
(в) CoVe-строка «Цены и история» показывает `N тикеров Tradernet · M yfinance (fallback)`
со статусом `warn`; (г) флаг `PRICE_FALLBACK=off|yfinance` (default off до валидации).
**Шаги:** интерфейс+обёртка (1д) → YF-провайдер+маппинг тикеров (2д) → ChainProvider+CoVe (1д)
→ тест-сверка Tradernet vs YF на 10 общих тикерах, |Δclose|<0.5% (1д) → зависимость в
requirements.txt/lock (pip-compile на linux/py3.11!) → включение флага в проде.
**Оценка: ~1 неделя.** Риск: KZ-бумаги (AIX) на Yahoo отсутствуют — fallback частичный by design.

## 2. Демо-режим бота (отчёт без счёта Freedom)

**Идея пользователя:** текстовая команда (напр. «демо» или «кэш») → полноценный отчёт по
фиктивному портфелю. Ценность: онбординг/продажа ДО подключения API-ключа.

**Архитектура:** демо — это **обычный прогон** `analyze_all()` на синтетическом
`Portfolio` (никаких веток в математике!):
```
tg_bot: /demo или текст «демо»
  → DEMO_PORTFOLIO (константа: 8–10 ликвидных US-тикеров + кэш, веса фикс.)
  → сборка positions-структуры ТОЧНО в формате freedom_portfolio.models.Portfolio
  → дальше существующий пайплайн без изменений; в отчёте бейдж «ДЕМО-ПОРТФЕЛЬ»
  → WhitelistMiddleware: демо доступно ДО подключения брокера (отдельная ветка гейта)
  → токены: демо бесплатно, но rate-limit 1/день на пользователя (иначе LLM-расход)
```
**Шаги:** демо-фикстура+конвертер (1д) → обход connect-требования во флоу (1д) →
бейдж в шаблонах BASE/DEEP + дисклеймер (0.5д) → лимит частоты (0.5д) → e2e-тест хендлера (1д).
**Оценка: ~4 дня.** Риск: путаница демо/реального отчёта → бейдж обязателен на КАЖДОЙ странице.

## 3. CDS-прокси: выключение без поломки Credit-пилляра

**Строгий анализ:** текущий сигнал — FRED HY-индекс + WGB суверенные, т.е. **не спреды
эмитентов**; в C-пилляр он входит бонусами/штрафами только `if cds_bps is not None`
(`scoring.py:244`). Функция `credit_score()` **уже** корректно деградирует до
SEC-компонент (Altman-Z, Piotroski-F, Interest Coverage) при отсутствии CDS — перераспределение
весов НЕ требуется, сумма пилляров и клипы не меняются.

**Решение (0 строк кода):** `CDS_DISABLED=1` в env бота (cloudbuild `--set-env-vars`).
CoVe честно покажет `CDS spreads · missing · CDS_DISABLED=1`; C-пилляр станет чисто
фундаментальным. **Рекомендация CTO:** выключить в проде ТОЛЬКО после сверки Buy/Sell-миграций
на снэпшоте (скрипт: прогнать scoring на текущем портфеле с/без CDS, задокументировать
изменившиеся action'ы в AUDIT). Если миграции приемлемы — включить флаг; удаление кода фида —
отдельным PR через 2 недели стабильности (мёртвый код не мешает, поспешное удаление — риск).

## 4. Lookback 730д → 1825 кал. (≈1260 торговых ≈ 5 лет) — ✅ ВКЛЮЧЕНО 2026-07-07

> **Единица измерения (важно):** параметр `days` в `freedom_portfolio/history.py` —
> **КАЛЕНДАРНЫЙ** (`start = today − timedelta(days=days)`). Поэтому «5 торговых лет
> ≈ 1260 ТОРГОВЫХ дней» = **1825 КАЛЕНДАРНЫХ** дней (не 1260 кал. — то было бы ~3.45 г).
> Дефолт `HISTORY_LOOKBACK_DAYS` теперь **1825** и совпадает со `SCENARIO_LOOKBACK_DAYS`
> — единое 5-летнее окно по всему проекту. Откат: `HISTORY_LOOKBACK_DAYS=730`.

**Что изменено (2026-07-07):**
- `investment_logic.get_market_data` default 730→**1825**; `history.get_candles/get_history_frame`
  default 730→**1825**; `services/fx_feed.DEFAULT_DAYS` 750→**1900** (FX-серия ОБЯЗАНА покрывать
  всё ценовое окно, иначе ранние даты мультивалютного портфеля не конвертируются → NaN →
  row-`dropna` усекает общее окно назад к 750);
- лейблы окна в `data_lineage`/`pdf_payload` стали ДИНАМИЧЕСКИМИ (читают env `HISTORY_LOOKBACK_DAYS`),
  чтобы CoVe-строка «Nd window» не расходилась с реальным конфигом;
- тест-пин обновлён (`test_lookback_env_bridge_defaults_to_1825`).

**Анализ репрезентативности (зачем 5 лет):** средний бизнес-цикл 5–7 лет; 730 кал. (~500 торг.)
не покрывали даже полфазы — Max DD/CVaR систематически **недооценивали хвосты** (окно
2024→2026 не видело медвежий рынок 2022 и цикл повышения ставок). 1825 кал. (~1260 торг.)
захватывает полный современный цикл: бычий 2021 → медвежий/ставки 2022 → восстановление
2023–24 → 2025–26. Это напрямую улучшает: (1) оценку хвостов CVaR/MaxDD; (2) стабильность
факторных бет (больше наблюдений на 10 факторов, κ-диагностика надёжнее); (3) бэктест-тир
(Панель B) получает статистику для hit-rate/fwd-returns. 10 лет (2520 торг.) сознательно НЕ
берём: retail-горизонт короче, а equal-weight на 10 годах тянул бы ковариацию устаревшими
режимами (доткомовская волатильность нерелевантна портфелю 2026).

**Анализ качества данных (риски и как они закрыты):**
1. **Молодые активы (IPO / KZ-AIX-листинги моложе 5 лет).** Движок делает row-level `dropna`
   на матрице лог-доходностей → общее окно = min-история среди ВСЕХ активов и фактор-ETF.
   Один молодой тикер усекает окно для всех. Смягчение (уже в коде): sparse-history дроп
   (`_build_portfolio_log_returns` → `return_coverage.dropped`, покрытый вес логируется) +
   `min_obs = max(2·k_factors, 30)` гейт против вырожденной регрессии. **Реактивность ядра
   сохраняется:** ковариация = EWMA(hl=63)⊕Ledoit-Wolf — последние ~3 месяца доминируют,
   поэтому расширение окна улучшает хвосты/беты, но НЕ «размывает» риск старыми режимами.
2. **Survivorship bias.** Тестируются/оцениваются только сегодня-живые тикеры — смещение
   вверх; для сценарного бэктеста явно вынесено в `DISCLAIMERS`.
3. **Покрытие Tradernet на 5 лет.** По US-мега-капам история полная; по KZ/AIX и молодым —
   может быть короче. Когда так — CoVe-строка честно покажет фактический возраст последнего
   close и (при sparse) дроп; отчёт не падает.
4. **FX-покрытие.** Поднято до 1900 кал. дней, чтобы 1825-дневное ценовое окно
   конвертировалось целиком (иначе скрытое усечение до 750).
5. **Стоимость.** Ковариация 1260×~25 инструментов — доли секунды/мегабайты, в 2Gi-лимит
   контейнера укладывается с запасом.

**Осознанное отклонение от исходного плана §4-б:** pairwise `.cov(min_periods=252)` в ОСНОВНОМ
движке НЕ вводили (он там был бы разрывом Do-No-Harm — меняет способ построения Σ для всех).
Pairwise-подход живёт в `scenario_engine` (свой изолированный слой), а основной движок
сохраняет исходный row-`dropna` контракт — это гарантирует, что расширение окна не меняет
*алгоритм*, только длину входного ряда. A/B-снэпшот 730 vs 1825 на реальном портфеле —
рекомендуется приложить в AUDIT при первом прод-прогоне.
