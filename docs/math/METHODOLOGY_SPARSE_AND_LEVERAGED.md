## 1. Анатомия проблемы

Портфель держит бумаги с разной глубиной истории:

| Класс | Пример | История | Что ломалось |
|---|---|---|---|
| Зрелые акции | AAPL, MSFT, NVDA | 5 лет (полное окно 1825 кал. дн) | — |
| Молодой листинг | FFSPC6.1028.AIX, CONL, XNDU | 60–250 торг. дней | Общее окно пересечения = возраст самого молодого → Sharpe/CVaR/MaxDD пересчитывались на суб-годовом окне |
| Сверхмолодой листинг | SPCX (< 60 торг. дней) | < 60 дней | Исключён из структурной модели (риск 0.00), но: (а) обнулял СОВМЕСТНОЕ окно спарклайнов → «нет истории»; (б) в Action Plan печатался «SELL» без количества и без объяснения |
| Плечевой daily-reset ETP | CONL (2×), XNDU | любая | Форвард β·μ игнорирует контрактный variance drag −½k(k−1)σ² → систематическое завышение прогноза |

Ключевое противоречие: **окно регрессии** (беты, факторная ковариация) и
**окно реализованных метрик** (Sharpe-числитель, CAGR, MaxDD, VaR/CVaR) — это
разные потребности. Регрессии нужна ОБЩАЯ матрица дат; реализованной метрике
нужна МАКСИМАЛЬНАЯ история книги. До F-22 оба сидели на одном (узком) окне.

## 2. Сравнение методов

### 2.1 Для книги со ступенчатыми историями (реализованные метрики)

| Метод | Суть | Плюсы | Минусы | Вердикт |
|---|---|---|---|---|
| Row-dropna (было) | Пересечение дат всех имён | Простота; одна матрица | Окно = самый молодой актив; Sharpe 12М невычислим; спарклайны пусты | ❌ причина бага |
| bfill (до F-6) | Дотянуть цены назад константой | «Полное» окно | Look-ahead: фиктивные нулевые доходности занижают σ и β | ❌ снято в F-6 |
| **Маскированный композит (ВЫБРАНО, F-15/F-22)** | r_port(t)=Σwᵢrᵢ1[торговался]/Σwᵢ1[торговался], день входит при покрытии ≥50% весов; кэш-долю сохраняем масштабом ×Σw | Вся история книги; без синтетики и look-ahead; конвенция композитных индексов (MSCI/GIPS-стиль); один ряд для KPI, кривой и таблицы периодов | Веса — сегодняшние (backcast-каveat, стандартен для композитов); дни <50% покрытия выпадают | ✅ |
| Stambaugh (1997) backfill | Достроить доходности молодого имени регрессией на длинные ряды (MLE) | Статистически эффективные моменты | Синтетические ряды в ОТЧЁТНОЙ метрике = чёрный ящик для клиента; сложная валидация | 🔜 не для отчётных KPI; кандидат для внутренних моментов |

### 2.2 Для структурной модели (беты/Σ) — roadmap

Сейчас: одна общая матрица дат; sparse-guard (<60 дн) исключает имя целиком.
Правильная Barra-практика — **пер-активные окна регрессии**: β каждой бумаги
оценивается на ЕЁ пересечении с факторами (факторные ETF всегда полные), а
факторная ковариация F — на ПОЛНОЙ истории факторов. Σ = B·F·Bᵀ + D тогда не
зависит от самого молодого имени вовсе. Дополнительно для окон 60–250 дней —
сжатие бет **Vasicek/Blume**: β̂* = w·β̂ + (1−w)·β_prior, w = n/(n+n₀), n₀≈120
(prior: 1.0 по Market, 0 по стилям; для ETP — заявленное плечо × β базового
актива). Это Спринт-план (см. §5): меняет Σ на ступенчатых книгах → требует
отдельной валидации, поэтому НЕ входит в hotfix («не ломай логику»).

### 2.3 Для плечевых daily-reset ETP (форвардная доходность)

Дневное лог-приближение k×-ETP: r_lev ≈ k·r_u − ½k(k−1)σ_u² — второй член
(**variance drag**) контрактный (ежедневная ребалансировка), НЕ шум. В
регрессии он оседает в свободном члене α̂. F-14 исключил α̂ из форварда для
всех имён (реализованный дрейф ≠ персистентная премия) — для ETP это
выплёскивает и drag → β·μ завышает прогноз (2× ETP при σ≈90% теряет ~20 пп/год
только на drag).

| Метод | Плюсы | Минусы | Вердикт |
|---|---|---|---|
| Игнорировать (было) | — | Систематическое завышение E[r] плечевых | ❌ |
| **Явная формула −½k(k−1)σ_u² + fee −ER/252 (ВЫБРАНО c P-1, 2026-07-13, для имён с известным k)** | Теоретически чисто; drag и cost-of-leverage раскрываются РАЗДЕЛЬНО (`leveraged_adjustments` в QC + integrity-чип); σ_u = σ_ETP/·k· из собственной истории ETP — база не загружается | Нужен реестр параметров (создан: `finance/leveraged.py`) | ✅ |
| **Удержание min(α̂,0) (F-23) — ФОЛБЭК для имён без параметров** | Drag берётся из ДАННЫХ (α̂ уже оценена); только вниз (положительная α никогда не добавляется — anti-218%-guard сохранён); обычные имена не тронуты | α̂ содержит и drag, и шум удачи (для ETP шум ⊂ консервативная сторона) | ✅ фолбэк |

Реестр: `finance/leveraged.py::LEVERAGED_ETP_REGISTRY` — {base: (L, базовый
актив, expense ratio)}; CONL = 2× COIN, ER 1.04%. Расширение без деплоя: env
`LEVERAGED_ETP_EXTRA=тикер1,тикер2` (детекция-только → α̂-фолбэк) и
`LEVERAGED_ETP_PARAMS="TICK:L[:UNDERLYING[:ER]];…"` (полные параметры).
Маршрутизация: `investment_logic.apply_leveraged_forward` — известный L+σ →
контрактный drag (α̂ игнорируется, без двойного счёта), иначе min(α̂,0).

### 2.4 Для сверхмолодых имён (SPCX, <60 дней)

Оценивать β по <60 точкам бессмысленно (шум), поэтому имя остаётся вне
структурной модели (sparse-guard F-6 без изменений — его вес честно разбавляет
риск). Изменена ПОДАЧА (F-20/F-21): (а) спарклайны больше не зависят от него —
серия KPI берётся из готового композита; (б) строка Action Plan несёт причину
«вне модели: история < 60 торг. дней» вместо голого SELL без количества.
Score-based действие сохраняется — pillar-скоринг работает на доступных данных
(Credit/Techn.), это осознанная политика.

## 4. Верификация формул Sharpe/доходности (аудит-статус)

Проверено на живом payload 2026-07-11 и синтетике: геометрическая аннуализация
`exp(mean_log·252)−1` (Itô-коррекция ✓), Sortino по Satchell (знаменатель по
ВСЕМ N, MAR=0 ✓), RFR геометрическая дневная и валютно-согласованная ✓,
Sharpe = (r_geo − rf)/σ_struct — смешанный базис задокументирован
(`sharpe_basis_note`, Integrity-панель) ✓. Ошибок в формулах нет; дефект был в
ОКНЕ данных, подаваемом в них (устранён F-22).
