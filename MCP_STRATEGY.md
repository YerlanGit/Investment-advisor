# MCP_STRATEGY.md — Стратегия перехода RAMP-бота на Model Context Protocol

> **Версия:** 2026-06-09 · Автор: Lead AI Architect
> **Контекст:** дополняет `AUDIT.md`. Фокус сохранён на **рве проекта** — математическое ядро (Эйлер-декомпозиция, Black-Litterman, Bootstrap-CVaR, выпуклые стресс-тесты) и адаптация под **KASE / Freedom Broker (Tradernet)**.
> **Тезис:** данные — коммодити, **аналитика и регион — ров**. MCP нужно использовать так, чтобы *потреблять* чужие данные и *экспонировать* свою математику.

---

## 0. TL;DR

| Направление | Что делаем | Эффект | Оценка |
|---|---|---|---|
| **MCP-клиент** | Тянем фундаментал/новости из готовых MCP-серверов | Шире данные меньшими силами | ~40–60 ч |
| **MCP-сервер** | Экспонируем `analyze_all`/CVaR/стресс как MCP-инструменты | Ров становится продуктом для агентов | ~60–80 ч |
| **News-MCP** | Реал-тайм новостной трекинг (News/Bloomberg MCP) | Динамический контекст + алерты | ~24–40 ч |
| **Скилы (Skills)** | Институциональная методология как Anthropic Skill | Консистентность нарратива, меньше токенов | ~16 ч |
| **Динамические стресс-тесты** | Code-execution / Python-sandbox MCP | Произвольные сценарии без релиза | ~40 ч |

## 1. Улучшение проекта: насколько MCP меняет архитектуру

Сейчас архитектура — **монолитный пайплайн**: `Движок (finance/*) → Адаптер (pdf_payload) → Шаблоны (v3)`, плюс один `messages.create` в `ai_narrative`. Данные (Tradernet/FRED/SEC-EDGAR) интегрированы кастомно, точка-в-точку.

MCP превращает это в **клиент-серверную модель с двумя ролями**:

```
                         ┌─────────────── ВНЕШНИЕ MCP-СЕРВЕРЫ ───────────────┐
                         │  Alpha Vantage · FMP · Financial Datasets · News  │
                         │  Bloomberg · FactSet  (фундаментал, котировки,    │
                         │  рейтинги, новости)                                │
                         └───────────────▲───────────────────────────────────┘
                                         │ (A) RAMP как MCP-КЛИЕНТ: тянет данные
                                         │
   ┌──────────────────── RAMP-БОТ ───────┴───────────────────────────────────┐
   │  finance/*  (MAC3-движок: Эйлер, BL, CVaR, стресс)  ← РОВ                 │
   │  pdf_payload (адаптер) → Jinja v3 → Telegram                             │
   │  ai_narrative (Claude: prompt-caching + structured outputs)             │
   └───────────────────────────────────┬─────────────────────────────────────┘
                                         │ (B) RAMP как MCP-СЕРВЕР: отдаёт аналитику
                                         ▼
                         ┌──────────── ПОТРЕБИТЕЛИ MCP ──────────────────────┐
                         │  Claude Desktop · корп-агенты · сам бот ·          │
                         │  будущий agentic-UX («спроси портфель»)            │
                         └───────────────────────────────────────────────────┘
```

### (A) RAMP как MCP-клиент — потребление данных
- **Что меняется:** кастомные коннекторы (`broker_api`, `fx_feed`, `sec_edgar`) дополняются/заменяются вызовами к готовым MCP-серверам через `mcp_servers` в Messages API или SDK-хелперы `anthropic.lib.tools.mcp` (`pip install anthropic[mcp]`).
- **Насколько сильно:** **умеренно** — это новый слой `data_providers/mcp_*`, движок не трогаем. Tradernet/KASE остаётся кастомным (нишевый, MCP-сервера нет).
- **Риск:** торговые/исполнительные MCP (Alpaca) — только за human-in-the-loop; новостные/фундаментальные — fence как `<untrusted_data>` (защита от prompt-injection уже есть в `ai_narrative._wrap_untrusted`).

### (B) RAMP как MCP-сервер — экспонирование рва
- **Что меняется:** новый процесс `mcp_server/` (отдельный Cloud Run сервис), который оборачивает `UniversalPortfolioManager.analyze_all`, `_bootstrap_cvar`, стресс-каталог, Black-Litterman в MCP-инструменты:
  - `compute_portfolio_risk(holdings) → {cvar, vol, sharpe, euler_trc[…]}`
  - `run_stress(holdings, shocks) → scenarios[…]`
  - `black_litterman(views) → target_weights[…]`
- **Насколько сильно:** **значимо, но изолированно** — это обёртка над существующими функциями (схемы + auth), движок не переписываем.
- **Почему это ров:** готовых MCP-серверов **институционального риск-разложения нет** (рынок MCP — это данные/исполнение). Ваша математика становится инструментом, которого нет у других агентов.

**Вывод:** архитектура из монолита превращается в **хаб**: движок в центре, данные втекают через MCP-клиент, аналитика вытекает через MCP-сервер. Это эволюция, а не переписывание — пайплайн «движок→адаптер→шаблон» сохраняется.

## 2. Преимущества для конечного инвестора

1. **Глубже и свежее данные** — фундаментал/рейтинги из нескольких источников (а не только SEC-EDGAR), без месяцев интеграций.
2. **Реал-тайм контекст** — нарратив учитывает свежие новости/даунгрейды по позициям (см. §3), а не только историю котировок.
3. **Динамические сценарии** — «а что если нефть −20% и тенге −15%?» считается на лету (см. §5), а не из фиксированного каталога.
4. **Портативность аналитики** — пользователь (или его корп-агент) может дёрнуть риск-движок RAMP из Claude Desktop: «посчитай CVaR моего портфеля» — без Telegram.
5. **Дешевле и быстрее** — prompt-caching + batch (уже внедрено/заложено) удешевляют генерацию; экономия может транслироваться в цену токена для пользователя.

## 3. Трекинг новостей в реальном времени (News / Bloomberg MCP)

**Да, MCP здесь — естественное решение.** Сейчас бот видит только историю цен (Tradernet) + статичные банковские PDF (ChromaDB RAG). News-MCP добавляет **живой поток**:

- **Подключение:** новостной MCP-сервер как `mcp_servers` коннектор. **Верифицировано (2026-06):** [Benzinga MCP](https://www.benzinga.com/apis/) — live, market-moving news; [Finnhub MCP](https://github.com/NimbleBrainInc/mcp-finnhub) — котировки+новости, есть SSE-стриминг; [Tiingo MCP](https://github.com/wshobson/tiingo-mcp) — 17 инструментов: новости, фундаментал, corporate actions. *Bloomberg MCP публично НЕ подтверждён — упоминать клиентам только после проверки enterprise-доступности.* Claude в DEEP-нарративе вызывает `get_news(tickers, since)` и вплетает свежие события в комментарий с цитатой-источником.
- **Алерты (новый сценарий продукта):** фоновый воркер опрашивает News-MCP по позициям пользователей; при материальном событии (даунгрейд, M&A, гайденс) — пуш в Telegram «⚠️ NVDA: даунгрейд от Morgan Stanley — пересчитать риск?».
- **Защита:** новости — `<untrusted_data>` (fence уже есть). Цитирование обязательно (`[News: source]`), как уже сделано для `[RAG: file]`.
- **Гибрид с RAG:** статичные банк-PDF (глубокая аналитика) + живые новости (свежесть) = полный спектр.

> Замечание о свежести данных vs обучении модели: новостной трекинг через MCP **снимает** ограничение knowledge-cutoff LLM — модель получает факты во время инференса, а не из весов.

## 4. Качество данных для прогнозов

| Сейчас | С MCP |
|---|---|
| Котировки: только Tradernet (риск пробелов по неликвиду) | + Alpha Vantage/FMP как фолбэк/кросс-валидация → меньше дыр в истории (закрывает H-1 edge-кейсы) |
| Фундаментал: SEC-EDGAR (US-only, лаг) | + Financial Datasets/FMP (шире покрытие, чаще обновление) |
| Макро: FRED | + консенсус-прогнозы, фьючерсы на ставку из data-MCP |
| Новости: нет | News-MCP (см. §3) |
| Кросс-валидация: нет | Несколько источников → детект расхождений (data-lineage уже есть — CoVe) |

**Эффект на прогнозы:** Black-Litterman качественнее с реальными консенсус-views (а не только score-derived); ковариация устойчивее на полной истории; режим-детект точнее со свежими макро-сигналами. **Важно:** прогнозную ответственность несёт **движок** (ваш ров), MCP лишь улучшает *вход*.

## 5. Сценарный анализ: динамические стресс-тесты через MCP-инструменты

Сейчас стресс-каталог **статичен** (7 параметрических шоков в `finance/stress.py`). MCP даёт два рычага:

1. **Произвольные пользовательские сценарии (Tool use):** инструмент `run_stress(holdings, custom_shocks)` (уже почти готов — параметр `scenario_shocks` в `analyze_all` существует, но не прокинут; см. AUDIT H-4-хвост). LLM переводит «нефть −20%, тенге −15%, ставка +100бп» в вектор шоков → движок считает с выпуклым ограничителем. Без релиза.
2. **Внешняя Python-песочница (Code Execution MCP / server-side code execution):** для сценариев, не покрытых факторной моделью (нелинейные опционные выплаты, Монте-Карло с кастомным распределением), Claude пишет и исполняет код в изолированной песочнице Anthropic, **не** трогая прод-движок. Результат возвращается как данные.

**Граница ответственности (критично для рва):** «штатные» институциональные сценарии (выпуклый cap ±35%, Politis-Romano CVaR) остаются в **вашем** детерминированном движке — это аудируемо и воспроизводимо. Песочница — только для *ad-hoc* исследований, помечается дисклеймером. Не отдавайте ядро риск-математики на откуп LLM-песочнице.

## 6. Маршрутизация моделей: Anthropic vs Google Gemini (честный разбор)

Задача нарратива — **не вычислять числа** (это делает MAC3-движок), а **интерпретировать и комментировать** уже посчитанные числа, не выдумывая и не пересчитывая их. Критичные качества: (1) **faithfulness/grounding** (не врать числами — ровно баг секторов 80.8%, который мы только что починили), (2) **instruction-following** (лимиты длины, теги-источники, «бери число ВЕРБАТИМ»), (3) надёжность **structured output**, (4) стоимость.

**Где Gemini объективно силён (честно):**
- **Контекст и мультимодальность:** крупнейшие контекстные окна, доминирование в мультимодальных задачах — релевантно, если в будущем кормить модели *картинки графиков* для верификации.
- **Structured output observability:** строгая JSON-schema валидация даёт меньше ошибок парсинга — но мы эту проблему уже закрыли на стороне Anthropic через forced `tool_choice`.
- **Экономика контекста / цена** на ряде задач конкурентна.
- **Чистое числовое многошаговое рассуждение:** на бенчмарке FinanceReasoning (238 сложных задач) лидирует GPT-5 (~88%), Gemini и Claude рядом — то есть для *вывода* новых чисел разрыв невелик.

**Где Claude сильнее именно для ЭТОЙ задачи (интерпретация, а не вычисление):**
- **Instruction-following и long-form writing** — лидерство Claude; для отчёта это ключ: соблюдение лимитов, тегов-источников, запрета на самостоятельную агрегацию.
- **GDPval-AA** (экономически ценные задачи: фин-моделирование/ресёрч) — Claude Sonnet 4.6 лидирует с большим отрывом.
- **Tool-augmented надёжность** — при использовании внешних инструментов Claude Opus 4.6 обходит Gemini 3.1 Pro (важно для MCP-маршрута §1).
- **Интеграционная зрелость** — бот уже на Anthropic, с injection-fencing, prompt-caching и structured outputs.

**Вывод по маршрутизации (без предвзятости):**
- **Корень проблемы «плохо описывает цифры» — не бренд модели, а ПАЙПЛАЙН.** Баг 80.8% был *grounding*-ошибкой: модели давали сырые сектора и позволяли свободно агрегировать. Мы починили это **provider-agnostic** способом (подаём authoritative `sector_complex` + запрет на самоагрегацию + structured outputs). Этот фикс работает с любой моделью.
- **Рекомендация:** оставить **Claude по умолчанию** (Haiku BASE / Sonnet DEEP; DEEP опционально → Opus 4.8 с adaptive thinking). Завести **тонкий provider-abstraction слой** (`NarrativeProvider` интерфейс) — тогда A/B Gemini на DEEP-нарративе стоит ~16–24 ч и даёт данные для решения. Но **не переносить ров в выбор провайдера**: дифференциация — математика и регион, а не марка LLM.
- **Когда Gemini оправдан:** если (а) стоимость на масштабе станет доминирующей, или (б) понадобится мультимодальная верификация графиков, или (в) самый длинный контекст. Тогда A/B через абстракцию — а не хардкод-замена.

> Реализация Gemini сейчас **не делалась** — это аналитическая рекомендация. Текущий пайплайн остаётся на Anthropic SDK; provider-abstraction — отдельная задача из roadmap.

## 7. Экономика и оценка внедрения

### 7.1 Трудозатраты (разработка)

| Эпик | Состав | Часы |
|---|---|---:|
| MCP-клиент (данные) | слой `data_providers/mcp_*`, 1–2 сервера (FMP/AlphaVantage), fencing, тесты | 40–60 |
| MCP-сервер (ядро) | `mcp_server/`: схемы для `analyze_all`/CVaR/стресс/BL, auth, Cloud Run, тесты | 60–80 |
| News-MCP + алерты | коннектор + фоновый воркер + Telegram-пуши + цитирование | 24–40 |
| Skills (методология) | упаковка институциональной методички как Anthropic Skill | 12–16 |
| Динамические стресс-тесты | прокинуть `scenario_shocks` + code-exec MCP + дисклеймеры | 32–40 |
| Provider-abstraction (A/B Gemini) | интерфейс `NarrativeProvider`, адаптер Gemini, eval-харнесс | 16–24 |
| **Итого (полный охват)** | | **184–260 ч** (≈ 5–7 недель одного инженера) |

> Поэтапно: «быстрые победы» (prompt-caching + structured outputs) **уже сделаны** в этом спринте (0 ч сверху). MCP-сервер ядра — highest-leverage, делать первым после данных.

### 7.2 Токены и инфраструктура (порядок величин)

Текущая стоимость нарратива на 1 отчёт (грубо): BASE Haiku 4.5 (~$1/$5 за 1M) + DEEP Sonnet 4.6 (~$3/$15 за 1M):

| Компонент | Без оптимизации | С prompt-caching | С caching + Batch (ночь) |
|---|---:|---:|---:|
| Системный промпт (~8K ток.) × 2 вызова | полная цена входа | **~0.1×** на повторах (TTL 5 мин) | + **−50%** к токенам |
| Вход (summary+RAG) | полн. | полн. | −50% |
| Выход (4.5K+7K ток.) | полн. | полн. | −50% |
| **Оценка/отчёт** | **~$0.18** | **~$0.15** (≈ −16% на системном префиксе) | **~$0.075** (ночной батч) |

- **Prompt caching (внедрено):** экономия растёт с объёмом (много пользователей в окне TTL читают системный префикс из кэша). Кэш per-model — BASE и DEEP кэшируются раздельно, оба выигрывают.
- **Batch (заложено):** −50% к токенам на плановых/ночных отчётах. Для 1000 отчётов/ночь: ~$75 vs ~$180 синхронно.
- **MCP-сервер ядра (инфра):** отдельный Cloud Run (min-instances=0 для on-demand) — десятки $/мес при низком трафике.
- **News-MCP:** подписка зависит от провайдера (Finnhub дешевле, Bloomberg — enterprise-ценник).
- **Structured outputs (внедрено):** не добавляет токен-косты; **снижает** потери от перегенерации сломанного JSON (раньше repair-проход на части вызовов).

### 7.3 Окупаемость
- **Спринт «быстрых побед» (сделан):** 0 доп. часов, мгновенная экономия токенов + детерминированный JSON (меньше фейлов отчёта → меньше возвратов токенов пользователям).
- **MCP-сервер ядра:** окупается как **продуктовая фича/дифференциация** (B2B-доступ к риск-движку), а не только экономией.
- **Batch:** окупается при ≥ нескольких сотен плановых отчётов/сутки.

## 9. Skills и Connectors: как меняется ЛОГИКА (не только архитектура)

MCP меняет *откуда берутся данные*; **Anthropic Skills** меняют *как бот думает*; **Connectors** меняют *что пользователь может подключить*.

- **Skills (методология как код).** Сейчас вся институциональная методичка (как трактовать CVaR, что значит HOTSPOT, как формулировать даунгрейд, дисциплина цитирования `[Quant Engine]/[RAG]`, режим-плейбук GS/Barclays/JPM) живёт в монолитном ~25 КБ системном промпте. Упаковка её в **Skill** (`SKILL.md` + reference-файлы, progressive disclosure) даёт: (1) меньший базовый промпт → дешевле и стабильнее кэш; (2) **версионируемость** методологии (откат, A/B); (3) on-demand загрузка только релевантных кусков (стресс-методичка грузится только для DEEP). Плюс готовые скилы `xlsx`/`pdf` — экспорт модели в Excel и чтение банковских PDF без кастома. **Логика меняется:** из «один большой промпт на всё» в «ядро + подгружаемые компетенции».
- **Connectors.** Для Telegram-бота прод-коннекторы (Slack/Drive/GitHub) второстепенны, НО Google Drive/Gmail-коннектор позволил бы инвестору «бросить» брокерскую выписку/письмо — бот распарсит и проанализирует. Главные коннекторы здесь — **финансовые MCP-серверы данных** (§1A) и **vaults** для их кредов с авто-refresh (секреты не попадают в песочницу).
- **Чистый эффект на логику:** пайплайн становится `данные(MCP) → сигналы(движок) → методология(Skill) → нарратив(Claude+structured) → доставка`. Движок остаётся центром (ров), но обрастает управляемыми, версионируемыми компетенциями.

## 10. Инсайдерские и официальные данные (новый класс сигналов)

> Всё ниже — **публичные раскрытия** (легально, не MNPI). Ценность не в самих данных (они у всех), а в том, что **ваш движок превращает их в детерминированные сигналы**, а LLM их *цитирует, а не выдумывает* (та же дисциплина SSOT, что и фикс секторов).

**Источники (через MCP / расширение `sec_edgar.py`):**
| Сигнал | Форма / источник | Свежесть | Как использовать |
|---|---|---|---|
| Сделки инсайдеров (CEO/CFO/директора) | **SEC Form 4** (Financial Datasets/FMP MCP, EDGAR) | ~2 раб. дня (свежо) | Кластер покупок на открытом рынке = бычий; отфильтровать авто-продажи 10b5-1 (шум) |
| Институциональные позиции (whales) | **Form 13F** | 45 дней (лаг!) | Smart-money позиционирование, «суперинвесторы»; использовать с поправкой на устаревание |
| Активистские доли | **Form 13D/G** (≥5%) | дни | Активист зашёл → событийный сигнал |
| Сделки конгресса/политиков | **STOCK Act** (Quiver Quant / Capitol Trades MCP) | дни–недели | Политический альфа-фактор (легально, публично) |
| Материальные события | **Form 8-K**, **Form 144** | реал-тайм | M&A, гайденс, смена топов, плановые продажи |
| Госстимулы и расходы | **FOMC** + dot-plot, **CHIPS Act / IRA**, USAspending, Treasury | реал-тайм–кварт. | Карта субсидий → секторальные tilts (полупроводники, чистая энергия, оборонка) |
| KZ-ниша (ров) | **Нацбанк РК** (ставка), **KASE** раскрытия, Самрук-Казына, нефте-фискальные потоки | дни | Региональный сигнал, которого нет у глобальных конкурентов |

**Конвейер обработки (качество, не сырьё):**
1. MCP/EDGAR → нормализация → **детерминированные скоры в движке**: `insider_cluster_score`, `congress_net_buy`, `institutional_flow_delta`, `gov_subsidy_exposure_flag`, `activist_present`.
2. Скоры втекают в три места: (a) **Black-Litterman views** (уверенность view ∝ сила×свежесть сигнала), (b) **альфа-фактор** в `scoring.py`, (c) цитируемый контекст нарратива.
3. **Анти-галлюцинация:** LLM получает посчитанный сигнал и цитирует `[Form 4]/[13F]/[Congress]`, не вычисляет сам — ровно как `sector_complex` после фикса.
- **Кейвиаты:** учитывать лаги (13F = 45д, Form 4 = ~2д), де-шумить (10b5-1), избегать look-ahead в бэктестах; для конгресс-данных — дисклеймер о точности раскрытий.

## 11. Прогнозные модели на качественных MCP-данных

MCP улучшает **вход**; модели живут в **вашем движке** (ров) и питают BL-views + scoring + стресс. Каждая модель — тестируемый источник альфы с измеримым IR (историю для walk-forward бэктеста тоже даёт MCP).

| # | Модель | Нужные данные (через MCP) | Куда втекает |
|---|---|---|---|
| 1 | **Insider-sentiment alpha** (кластер Form 4) | Form 4 история | BL-view / фактор |
| 2 | **Smart-money flow** (Δ13F, 13D) | 13F/13D история | tilt-фактор |
| 3 | **Congressional-trade factor** | STOCK Act | альфа-фактор |
| 4 | **PEAD / earnings drift** (сюрприз → пост-отчётный дрейф) | консенсус EPS + факты (FMP) | моментум-фактор |
| 5 | **Analyst-revision momentum** (пересмотры консенсуса) | revisions-feed | фактор |
| 6 | **News-sentiment factor** (NLP-скор новостей) | News MCP | кратко-горизонтный tilt |
| 7 | **Macro-nowcast → вероятностный режим** | высокочастотный макро (FRED+) | апгрейд `regime.py`: дискретный → вероятностный |
| 8 | **Implied-vol / options-skew** (форвард-риск) | options-chain MCP | заменить/дополнить историч. vol; богаче стресс |
| 9 | **CDS-implied default** (рыночный кредит) | `cds_feed` (есть!) + spreads | усилить Altman-Z |
| 10 | **FX/нефть-тенге** (региональный ров) | oil + KZT + sovereign | кросс-валютный риск KZT |
| 11 | **Policy-tailwind exposure** (IRA/CHIPS/оборонка) | USAspending + программы | секторальный policy-скор |

**Принцип:** не «LLM предсказывает», а «движок считает фактор → BL/scoring агрегирует → LLM объясняет». Это аудируемо, бэктестируемо и усиливает ров, а не размывает его в промпт.

## 12. Движок генерации идей (idea-quality) — от «AI-пиков» к «quant-screened, AI-explained»

Сейчас `stock_picks` генерит LLM из скоров. Апгрейд — **data-driven, ранжированный, бэктестируемый** конвейер (та же дисциплина, что фикс секторов: подаём authoritative-числа, LLM объясняет):
1. **Юниверс** из MCP (ликвидные, mandate-eligible кандидаты).
2. **Композитный альфа-скор** на каждого кандидата в движке (факторы §11 + инсайдер + ревизии + новости + policy).
3. **Ранжирование**; LLM объясняет top-K, **цитируя** посчитанные сигналы, не выдумывая ранг.
4. **Constraint-aware:** мандат, лимиты концентрации, turnover-cap (уже есть в BL), налоги.
5. **Бэктест правил** на истории MCP → публиковать hit-rate / IR идей прямо в отчёте (аудит + доверие).
6. **Персонализация** под профиль и текущие позиции (логика замены held-picks уже есть).
- **Эффект:** идеи становятся защитимыми и согласованными с институциональным рвом, а не «креативом модели».

| Эпик (новый) | Состав | Часы | Данные/инфра |

## 14. Верификация документа и протокольная зрелость MCP (ревизия 2026-06-10)

> Полная перепроверка фактов документа против актуального состояния экосистемы. **Код не менялся** — это аналитическая ревизия перед внедрением.

### 14.1 Что подтверждено
- **Цены/кэш/батч (§5, §7, §13):** Haiku 4.5 $1/$5 и Sonnet 4.6 $3/$15 за 1M; cache-read ≈0.1×, cache-write 1.25× (5-мин TTL); мин. кэшируемый префикс Haiku 4096 / Sonnet 4.6 2048 токенов; Batch API −50% — всё актуально.
- **API-механика (§1):** `mcp_servers` (MCP-коннектор Messages API), SDK-хелперы `anthropic[mcp]`, vaults для MCP-кредов с авто-refresh (credentials никогда не попадают в песочнику; инъекция Anthropic-side proxy) — подтверждено по актуальному референсу.
- **News-MCP (§3):** провайдеры верифицированы и уточнены — Benzinga (live), Finnhub (SSE-стриминг), Tiingo (news+fundamentals+corp actions). ⚠️ **Поправка:** Bloomberg MCP публично не подтверждён — из обещаний клиентам убрать до проверки.

### 14.2 Зрелость протокола — что это значит для НАШЕГО MCP-сервера (§1B)
- **Текущая стабильная ревизия спеки: `2025-11-25`.** Анонсирован **RC `2026-07-28`** — крупнейшая ревизия протокола (6 SEP по авторизации). Наш сервер математического ядра строим против `2025-11-25`, с адаптационным слоем под RC-изменения (они почти целиком в auth).
- **Авторизация = OAuth 2.1 уже обязательна:** PKCE для клиентов (OAuth 2.1 §7.5.2), **RFC 8707 Resource Indicators** (токены привязаны к конкретному серверу: клиент ОБЯЗАН слать `resource`, сервер ОБЯЗАН валидировать аудиторию), RC добавляет RFC 9207 `iss`-валидацию и `application_type` при DCR. → дизайн-требования к нашему серверу: resource-bound токены, никакого passthrough чужих токенов.
- **Дистрибуция:** официальный **MCP Registry** — канал дискаверибилити для нашего сервера; в роадмапе **Server Cards** (`/.well-known/`) — заложить эндпоинт метаданных заранее.
- **Институциональный аргумент:** по MCP уже существует **гайданс NSA (CSI)** по безопасному развёртыванию — для банковских/энтерпрайз клиентов наличие протокольного security-гайдлайна от регулятор-грейд органа упрощает комплаенс-разговор.

### 14.3 Security-чеклист интеграции (расширение §1A-рисков)
| Угроза | Митигция в нашем контексте |
|---|---|
| **Tool poisoning / rug-pull** стороннего MCP-сервера (описание инструмента меняется после ревью и инжектит инструкции) | Пин версии сервера; снапшот tool-описаний в репо + diff-алерт при изменении; allowlist конкретных инструментов, не `*` |
| Prompt-injection через **данные** MCP (новости/фундаментал) | Уже решено для RAG: `<untrusted_data>`-fence + цитирование — распространить на ВСЕ MCP-выводы |
| Кросс-серверная утечка токенов | RFC 8707 resource-binding (см. 14.2); креды только в vaults, не в env песочницы |
| Schema-абьюз (malformed/oversized входы к НАШЕМУ серверу) | Strict JSON-schema валидация входов `compute_portfolio_risk`/`run_stress` + лимиты размеров (паттерн уже есть в structured outputs) |
| Экзекьюшн-серверы (Alpaca и т.п.) | Не подключать к LLM без human-in-the-loop (повтор §1A) — торговые действия только через подтверждение в Telegram |

### 14.4 Итоговая поправка к роадмапу (§8)
Порядок не меняется; добавляются два пункта подготовки: (а) при старте эпика «MCP-сервер ядра» — сразу OAuth 2.1 + resource-binding по `2025-11-25` и заявка в MCP Registry; (б) перед продакшеном — прогон security-чеклиста 14.3 и фиксация tool-описаний потребляемых серверов в репо.
